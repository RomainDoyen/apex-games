Données — Vue d’ensemble

Supabase RAWG Dernière mise à jour : 2026-02-15

Sources de données

  • RAWG API : catalogue de jeux (tendances, recherche, filtres, détails). Données en lecture seule, pas de persistance côté app.
  • Supabase : authentification (utilisateurs, sessions), backlog (colonnes, jeux par utilisateur), et éventuellement cache métier côté backend.

Backend (NestJS)

Le backend utilise des modules dédiés : AuthModule (Supabase Auth), BacklogModule (Supabase ou tables liées), RawgModule (appels RAWG), GameCacheModule (cache optionnel des jeux). La configuration (URL, clés) se fait via variables d’environnement.

Frontend

Les favoris sont persistés en local (Zustand + localStorage) ; le backlog est synchronisé avec le backend (donc Supabase). Voir Favoris et Backlog.

Base de données (Supabase)

Apex Games n’utilise pas de script SQL dans le dépôt : la base est hébergée et gérée par Supabase. Le schéma (tables, RLS, politiques) se configure dans le dashboard Supabase ou via des migrations Supabase.

  • Auth : utilisateurs et sessions gérés par Supabase Auth (pas de table users à maintenir à la main).
  • Backlog : une ou plusieurs tables (ex. backlog ou backlog_games) pour stocker les jeux par utilisateur et par statut (à jouer, en cours, terminé). À documenter dans donnees/bdd.md quand le schéma est stabilisé.
  • Cache / autres : si des tables métier sont ajoutées (cache jeux, préférences), les décrire dans donnees/bdd.md et mettre à jour cette page.

Référence écrite du schéma (pour les développeurs) : donnees/bdd.md.
